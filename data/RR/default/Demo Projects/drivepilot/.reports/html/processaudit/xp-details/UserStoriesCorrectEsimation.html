How To Fix Violation
If rule score is N/A, then fulfill precondition - existence of user stories in unresolved state and existence of developer for project. To ensure this check that rules RP 1.2 - Create user stories and PO 1.2 - Create fully staffed team are passed. Also check that custom field Risk is defined in Polarion configuration for work item with Requirement type (that represents user stories). Follow step-by-step instructions how to fix violation:
    1. Switch to the Projects perspective
    2. Select Work Items in the Topics tree
In order to fix violated user story make following:
    1. Select violated user story
    2. Click on the Edit button
    3. If user story estimation is missed then enter estimation from 5 to 15 days to the Initial Estimate field
    4. If risk is missed then enter risk value to the Risk field
    5. If estimation is less then allowed, then extend user story functionality scope, edit Description field correspondently and update estimation in the Initial Estimate field. If estimation is more then allowed, then narrow user story functionality scope, edit Description field correspondently and update estimation in the Initial Estimate field.
    6. Click on the Save button
XP Theory
As soon as user stories were created by customers, they should be processed by development team. Developers estimate how long the stories might take to implement in terms of ideal development weeks. Ideal means that there is absolutely nothing else to do. No dependencies, no extra work, no other type of assignments, but do include tests. Each user story should get from 1 to 3 weeks estimation. If estimation is more or less it means that user stories should be split or combined. Together with estimation development team defines how risky is their estimate. Risk value can be used to identify user stories, which require technical aspects proof by spike solutions creation.

Instructions
All work items, which represent user stories, should be estimated by developers, i.e. Polarion users with project_developer identifier role. In the Polarion user story estimation is stored as the Initial Estimate value of correspondent work item. Acceptable estimation should be not less then 5 days and not more then 15 days, otherwise user story should be merged with other or split. Together with estimation risk value for user story should be set. In the Polarion risk is as the Risk filed value of the work item.

How it works
This rule is automatically checked. During the check, all unresolved user stories are verified. If user story without estimation or risk is found then new violation is created and assigned to user story assignee. If estimation is defined, but it was set not by user with project_developer identifier role, then new violation is created and assigned to the project manager. If estimation is set, but doesn't fit to required bounds, then new violation is created and assigned to user story assignee.
