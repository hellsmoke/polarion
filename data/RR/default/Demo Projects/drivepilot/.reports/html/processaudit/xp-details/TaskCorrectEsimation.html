How To Fix Violation
If rule score is N/A, then fulfill precondition - existence of current iteration and work items of type Task, which supposes that there are unresolved tasks, assigned to iteration. To ensure this check that rules IP 1.2 - Create tasks and IP 1.1 - Define iteration are passed.
Follow step-by-step instructions how to fix violation:
    1. Switch to the Projects perspective
    2. Select Work Items in the Topics tree
In order to fix violated task make following:
    1. Select violated task
    2. Click on the Edit button
    3. If task estimation is missed then enter estimation from 1 to 24 hours to the Initial Estimate field
    4. If estimation was made not by assignee, then assignee should change estimation
    5. If estimation is less then allowed, then extend task functionality scope, edit Description field correspondently and update estimation in the Initial Estimate field. If estimation is more then allowed, then narrow task functionality scope, edit Description field correspondently and update estimation in the Initial Estimate field.
    6. Click on the Save button
XP Theory
When user stories are split to tasks, development team defines estimation for tasks. It's important that estimation should be set by developer, who assigned to the task. People are not interchangeable and the person who is going to do the task must estimate how long it will take. Estimation should be set in the predefined bounds. Tasks, which exceed estimation bounds, should be split or merged.

Instructions
All work items, which represent tasks, should be estimated by assignee. In the Polarion task estimation is stored as the Initial Estimate value of correspondent work item. Acceptable estimation should be not less then 1 hour and not more then 24 hours, otherwise task should be merged with other or split.

How it works
This rule is automatically checked. During the check, all unresolved tasks, assigned to current iteration, are verified. If task without estimation is fond then new violation is created and assigned to task assignee. If estimation is defined, but it was set not by assignee, then new violation is created and assigned to the assignee. If estimation is set, but doesn't fit to required bounds, then new violation is created and assigned to task assignee.
