XP Theory
Projects are developed iteratively, so a time to the next release is split to iterations. Iteration is general cell of the project life cycle during which new functionality added to project. At the same time during iteration project passes through micro cycles of planning, analysis, development, testing and delivery, which allow keeping project in optimal state for all these areas. From this point of view iterations duration should be from one to few weeks in order to have a time to add new features and at the same time keep a short cycles, which can be controlled easily. When team defines comfortable iteration duration, it should be kept unchanged over the time of project development. Equal duration of iterations allows easy team performance tracking and next tasks planning based on known performance values.

Instructions
During upcoming release planning project manager should care about splitting release to iterations with equal duration, which is in defined bounds from 5 to 15 days. In the Polarion iterations are represented by timepoints. Iteration is a time interval between two neighbor timepoints. Iterations which belong to release should't have special identifier, which starts from Release. Last release iteration is a timepoint which should have special identifier, which starts from Release Preconditions for this rule is existence of current release and iteartions for the current release.

How it works
This rule is automatically checked. During the check, open timepoints from the current release, which represent iterations, are analyzed. Timepoints considered as open if they have Closed property set to false. Duration of these iterations is analyzed and if it is out of required bounds, then new violation is created and assigned to the project manager. Also iterations duration is compared to each other and if some iterations have different from majority duration then new violations are created for them and assigned to the project manager.
