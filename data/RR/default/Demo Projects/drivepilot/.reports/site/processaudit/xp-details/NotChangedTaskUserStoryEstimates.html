How To Fix Violation
If rule score is N/A, then fulfill precondition - existence of iteration plan. To ensure this check that rule IP 2.1 - Create iteration plan is passed.
Due to origin of this rule, violations can't be directly resolved. Instead team should avoid re-estimation of tasks and user stories, assigned to current release. The score will be improved as soon as violated items will be resolved or iteration is passed.
XP Theory
User stories and tasks estimation should not be changed, even if during their implementation become clear that estimation was wrong. If estimation will be changed, it causes the changes in velocity calculation and other areas related with planning, which rely on consistent estimates. Instead, if wrong estimate kept in original step, it can be take into account during next iteration planning through velocity calculation. If the team requires changing estimations, it should be done not for particular user story or task, but for all items, including release and iteration plans changes. Only in this case estimation changes can be consistent.

Instructions
Project team should avoid changing estimations for user stories and tasks, assigned to current iteration, because it can break velocity calculator and planning. It's allowed to change estimation only for unassigned items.

How it works
This rule is automatically checked. During the check, all unresolved user stories and tasks, assigned to the current iteration, are verified. If their estimation was changed since assigning to iteration, then new violation is created and assigned to the project manager, who is responsible to control this rule.
