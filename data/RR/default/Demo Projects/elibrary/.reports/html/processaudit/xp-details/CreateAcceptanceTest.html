How To Fix Violation
If rule score is N/A, then fulfill precondition - current release is planned and existence of resolved user stories, assigned to current release. To ensure that current release is planned check that rule RP 1.1 - Define release is passed. If project just recently started, it's a normal situation that the score is N/A.
In order to fix violation team should create acceptance tests for resolved user stories, assigned to the current release. Follow step-by-step instructions how to fix violation:
    1. Switch to the Projects perspective
    2. Select Work Items in the Topics tree
If there are no acceptance tests for user story, they should be created:
    1. Click on the Create button and select Test Case from the popup menu
    2. Fill required fields
    3. Click on the Create button
In order to link user story and acceptance test make following:
    1. Select acceptance test in list of work items
    2. Click on the Edit button
    3. Select Requirement in the Linked Work Items > Title area and select relates to role
    4. Click on the Add button to add a link
    5. Click on the Save button
XP Theory
Acceptance tests are created from user stories. User stories assigned to iteration are translated into acceptance tests during the iteration. The customer specifies scenarios to test when a user story has been correctly implemented. A story can have one or many acceptance tests, what ever it takes to ensure the functionality works. A user story is not considered complete until it has passed its acceptance tests.

Instructions
User story can be marked as completed only if it passes acceptance testing. Acceptance testing is performed using a set of acceptance tests, created during iteration for each planned user story. In Polarion acceptance tests are stored as work items with Test Case type, which are linked with correspondent user story. Thus, acceptance tests should have relates to link role to user story work item.

How it works
This rule is automatically checked. During the check, all resolved user stories, assigned to current release, are verified. If there are resolved user stories, which have no linked acceptance tests, then new violations are created and assigned to user story author, who is responsible for acceptance tests creation.
