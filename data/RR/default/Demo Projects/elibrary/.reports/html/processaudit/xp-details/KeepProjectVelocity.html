How To Fix Violation
If rule score is N/A, then fulfill precondition - iteration plan existence for previous and current iterations. To ensure that current iteration is planned check that rule IP 1.1 - Define iteration is passed.
In order to fix violation current iteration plan should be changed to align total estimation of assigned tasks and defects to the project velocity from previous iteration. Project manager should assign more or unassign some user stories and correspondent tasks from current iteration.
Follow step-by-step instructions how to fix violation:
    1. Switch to the Projects perspective
    2. Select Work Items in the Topics tree
In order to assign/unassign work items from iteration make following:
    1. Select work item in list of work items
    2. Click on the Edit button
    3. To unassign task/defect from the iteration select empty value in the Timepoint combo-box
    4. To unassign user story from the iteration select current release in the Timepoint combo-box
    5. To assign task, defect or user story to the current iteration select it in the Timepoint combo-box
    6. Click on the Save button
XP Theory
The project velocity (or just velocity) is a measure of how much work is getting done on the project. This measurement is very useful for iteration planning. The rule is simple - plan for the current iteration the same amount of work, which was completed in the previous iteration. It makes project more predictable, because usually development team works with the stable speed. In order to calculate velocity you should summarize estimation for all completed tasks on the previous iteration. For the current iteration you should select tasks with total estimation equal to velocity value. This simple mechanism allows developers to recover and clean up after a difficult iteration and averages out estimates. The project velocity goes up by allowing developers to ask the customers for another story when their work is completed early and no clean up tasks remain.

Instructions
Before creation of the iteration plan, project manager should calculate project velocity by summarizing estimations for tasks and defects work items, which were completed during previous iteration. Then customers one-by-one select user story to add into the iteration plan. Developers split this user story to tasks, define estimates and add estimates to the total estimation for the iteration. Project manager should care to get total iteration estimation close to the velocity value as possible, but not more then velocity. Precondition for this rule is existence of the iteration plan for current and previous iterations.

How it works
This rule is automatically checked. During the check, velocity from previous iteration is compared with total estimation for current iteration. Velocity is calculated as summary of resolved task and defect work items estimates, assigned to previous iteration. Total estimation for current iteration is calculated as summary of task and defect work items estimates, assigned to current iteration. If total estimate for current iteration is more or less previous iteration velocity on more than configured difference threshold, then violation is create assigned to project manager, which is responsible for iteration planning.
