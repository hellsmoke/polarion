How To Fix Violation
If rule score is N/A, then fulfill precondition - existence of automatic tests. To ensure this check that rule QC 1.1 - Create automatic tests is passed.
Violations for this rule can't be fixed directly, because we can't change the date of source code and tests creation. But the overall score can be improved if the team will follow the practice to create tests before source code for new code, added to the project.
XP Theory
One of the corner stones of the automatic testing is a requirement to write tests before the code. When developers start work on some task they first of all create tests, which tests behavior of classes and methods, which will be implemented. It's pretty simple - requirements are translated to tests, which should verify them and after that classes and methods are implemented to pass these tests.

Instructions
When developers start creating new code, they should create finished tests for the code and then create the code itself. Tests are considered as created before the code if they are committed to the repository earlier as the code, which was tested.

How it works
This rule is automatically checked. During the check, automatic tests execution results are evaluated. For each package, which has direct classes, percent of public methods written before source code is calculated. If this percent is less then 50, then violation is created and assigned to project manager, who is responsible for the quality of automatic testing.
