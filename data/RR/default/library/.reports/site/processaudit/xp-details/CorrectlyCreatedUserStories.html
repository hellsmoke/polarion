How To Fix Violation
If rule score is N/A, then fulfill precondition - existence of user stories in unresolved state and existence of a customer for project. To ensure this check that rules RP 1.2 - Create user stories and PO 1.2 - Create fully staffed team are passed.
Violations caused by the fact that user stories were created not by customers can't be fixed directly, because it's not possible to change first author of the user story. Nevertheless you are able to improve the score if you will follow the rule during next user stories creation. Moreover, score can become 100% when violated user stories will be resolved and all new follow the rule requirements.
Violations, caused by too short or too long description can be fixed by changing description to appropriate.
Follow step-by-step instructions how to fix violation:
    1. Switch to the Projects perspective
    2. Select Work Items in the Topics tree
In order to fix violated user story make following:
    1. Select violated user story
    2. Click on the Edit button
    3. Edit Description field
    4. Click on the Save button
XP Theory
One of the key aspects of project organization is direct participation of customers in project creation. Customers have business area knowledge and can formulate project requirements, thus they should create user stories. User stories should avoid specific technical language and should use customer's terminology instead. The only purpose of user stories is providing enough detail to make a reasonably low risk estimate of how long the story will take to implement, so usually they are in the format of about three sentences of text.

Instructions
All user stories should be created by customers, i.e. Polarion users with project_customer identifier role, and stored in Tracker as work items with Requirement type. User story text is stored in Description field and should not be either too long or too short - it should be between 20 and 300 characters length.

How it works
This rule is automatically checked. During the check, authors for unresolved user stories are analyzed and if they have no project_customer identifier role, then new violation is created and assigned to project manager. Description filed for unresolved user stories is also analyzed and if it contains text shorter then 20 and longer then 300 symbols, then new violation is created and assigned to author of the user story.
